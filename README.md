# Introdução

O avanço das tecnologias digitais tem transformado de maneira significativa a forma como serviços são oferecidos e consumidos em diferentes setores da sociedade, inclusive no ambiente universitário. Nesse contexto, a busca por soluções tecnológicas que promovam maior praticidade, agilidade e eficiência no atendimento torna-se cada vez mais necessária, especialmente em espaços que recebem grande fluxo de pessoas em períodos curtos de tempo, como é o caso das cantinas instaladas em instituições de ensino superior.

Considerando essa realidade, o presente projeto extensionista propõe o desenvolvimento do aplicativo multiplataforma SonhoDoce, idealizado como uma ferramenta de apoio à modernização do atendimento em uma cantina universitária. O estabelecimento em questão atende diariamente um público diversificado, formado majoritariamente por estudantes de graduação e pós-graduação, além de professores, colaboradores e funcionários administrativos da instituição.

Por estar inserida em um ambiente acadêmico dinâmico, a cantina convive com momentos de alta demanda concentrados principalmente nos intervalos das aulas, nos horários de almoço e nas trocas de turno. Essa característica gera desafios operacionais importantes, tais como a formação de filas, a sobrecarga no atendimento presencial, a dificuldade de organização do fluxo de clientes e a limitação no acesso rápido às informações sobre os produtos oferecidos.

Além disso, a utilização de cardápios físicos impressos ou painéis fixos apresenta limitações que interferem diretamente na eficiência do serviço, como a dificuldade de atualização de preços, a pouca flexibilidade na inclusão ou remoção de produtos e a comunicação ineficiente de itens indisponíveis.

Diante desse cenário, o aplicativo SonhoDoce surge como uma proposta de intervenção tecnológica voltada à melhoria do atendimento e da comunicação entre a cantina e seu público, oferecendo um cardápio digital acessível, interativo e atualizado em tempo real.

# Lógica do Sistema 

O desenvolvimento do aplicativo SonhoDoce foi estruturado com base em princípios de organização modular e separação de responsabilidades, utilizando o framework Flutter para construção da interface e o Firebase como serviço de backend, especialmente para autenticação de usuários.

A lógica do sistema inicia-se na camada de entrada da aplicação, representada pelo arquivo main.dart, responsável por inicializar o Firebase e definir a tela inicial do sistema. A partir desse ponto, o fluxo de execução é direcionado para a interface de autenticação, onde o usuário pode optar por realizar login, cadastrar-se ou recuperar sua senha.

O processo de autenticação é gerenciado por uma camada de serviço denominada AuthService, que encapsula as chamadas ao Firebase Authentication. Essa camada abstrai a lógica de comunicação com o backend, permitindo que as telas da aplicação interajam com métodos como signInWithEmailAndPassword, createUserWithEmailAndPassword e sendPasswordResetEmail de forma simplificada e organizada.

Após a autenticação bem-sucedida, o sistema redireciona o usuário para a página inicial (HomePage), que funciona como ponto central de navegação. Nessa tela, o usuário pode acessar as categorias de produtos disponíveis, sendo essa organização fundamental para melhorar a experiência e reduzir o tempo de busca por itens.

A listagem de produtos é construída a partir de uma estrutura de dados previamente definida (ProductModel), que contém atributos como nome, descrição, preço, categoria e imagem. Esses dados são carregados por meio de uma lista mock (mockProducts), que simula um banco de dados durante a fase de desenvolvimento.

Quando o usuário seleciona uma categoria, o sistema executa um filtro na lista de produtos, retornando apenas os itens correspondentes. Esse processo utiliza funções de filtragem sobre listas, demonstrando o uso de lógica condicional e manipulação de coleções em Dart.

Ao selecionar um produto, o usuário é direcionado para a página de detalhes (ProductDetailPage), onde são exibidas informações completas do item e disponibilizada a opção de adicioná-lo ao carrinho.

A gestão do carrinho é controlada por uma camada lógica (CartController), responsável por armazenar, atualizar e remover itens selecionados. Essa estrutura segue o padrão de gerenciamento de estado, garantindo que as alterações realizadas no carrinho sejam refletidas corretamente na interface.

Na etapa final, o usuário pode acessar a página de checkout (CheckoutPage), onde ocorre a simulação do processo de finalização do pedido. Embora ainda não haja integração com sistemas de pagamento, a estrutura foi preparada para futuras expansões.

De forma geral, o sistema foi desenvolvido seguindo uma arquitetura simples, porém eficiente, baseada na separação entre interface (UI), lógica de negócio (controllers/services) e modelos de dados (models). Essa organização facilita a manutenção, escalabilidade e evolução do aplicativo, permitindo a inclusão de novas funcionalidades, como integração com banco de dados real e sistemas de pagamento.

# Arquitetura do Sistema

O sistema SonhoDoce foi desenvolvido com base em uma arquitetura em camadas, visando organização, escalabilidade e facilidade de manutenção. Essa arquitetura permite a separação clara entre responsabilidades, tornando o código mais estruturado e compreensível.

A aplicação está dividida em três camadas principais:

* Camada de Apresentação (UI):
Responsável pela interface com o usuário, composta pelas páginas desenvolvidas em Flutter, como login, cadastro, página inicial, categorias, produtos, carrinho e checkout. Essa camada gerencia a interação do usuário com o sistema.

* Camada de Lógica de Negócio:
Representada pelos controllers e serviços, como o AuthService e o CartController, responsáveis por processar as regras do sistema, gerenciar estados e controlar o fluxo de dados entre interface e backend.

* Camada de Dados:
Composta pelos modelos (models), como ProductModel e CartItemModel, além da integração com o Firebase Authentication, responsável pela persistência e validação das informações de usuários.

Essa estrutura garante maior modularidade, permitindo a evolução do sistema com a adição de novas funcionalidades, como banco de dados em tempo real e integração com sistemas externos.

# ANÁLISE DE REQUISITOS DO SISTEMA

1. Introdução à Análise de Requisitos

A análise de requisitos representa uma etapa essencial no desenvolvimento de sistemas, pois estabelece de forma estruturada as funcionalidades e características que a aplicação deve possuir para atender às necessidades identificadas no contexto real. No presente projeto, essa etapa foi conduzida a partir da observação do funcionamento de uma cantina universitária, considerando aspectos operacionais, fluxo de usuários e limitações do modelo tradicional de atendimento.

Com base nessa análise, foi proposto o desenvolvimento do aplicativo SonhoDoce (Doce Campus), com o objetivo de digitalizar o acesso ao cardápio, melhorar a organização do atendimento e proporcionar maior eficiência na interação entre usuários e estabelecimento.

A definição dos requisitos foi realizada de forma alinhada à solução implementada, garantindo coerência entre o problema identificado, os objetivos do projeto e as funcionalidades desenvolvidas. Além disso, buscou-se estruturar os requisitos de maneira clara e organizada, permitindo não apenas a compreensão do sistema, mas também sua evolução futura.

2. Requisitos Funcionais

Os requisitos funcionais descrevem as operações que o sistema deve realizar, representando diretamente as ações disponíveis ao usuário.

* RF01 — Autenticação de usuários

O sistema deve permitir que o usuário realize login utilizando e-mail e senha, garantindo acesso seguro às funcionalidades da aplicação.

* RF02 — Cadastro de usuários

O sistema deve possibilitar a criação de contas por meio do preenchimento de dados como nome, e-mail, telefone e senha.

* RF03 — Recuperação de senha

O sistema deve permitir a recuperação de senha por meio do envio de instruções ao e-mail cadastrado.

* RF04 — Acesso à página inicial

Após autenticação bem-sucedida, o usuário deve ser redirecionado automaticamente para a página principal do sistema.

* RF05 — Visualização de categorias

O sistema deve disponibilizar categorias de produtos, permitindo uma navegação estruturada e intuitiva.

* RF06 — Listagem de produtos

O sistema deve apresentar os produtos disponíveis de acordo com a categoria selecionada pelo usuário.

* RF07 — Visualização de detalhes do produto

O sistema deve exibir informações detalhadas dos produtos, incluindo nome, descrição, categoria e preço.

* RF08 — Adição de produtos ao carrinho

O sistema deve permitir que o usuário adicione produtos ao carrinho de compras.

* RF09 — Gerenciamento do carrinho

O sistema deve permitir a visualização dos itens selecionados, bem como a alteração de quantidades, remoção de produtos e cálculo do valor total.

* RF10 — Finalização do pedido

O sistema deve possibilitar a simulação da finalização do pedido, apresentando um resumo das escolhas do usuário.

* RF11 — Tela de informações do aplicativo

O sistema deve disponibilizar uma tela contendo informações sobre o projeto, contexto acadêmico e desenvolvedor.

3. Requisitos Não Funcionais

Os requisitos não funcionais definem as características de qualidade do sistema, influenciando diretamente sua eficiência, segurança e usabilidade.

* RNF01 — Multiplataforma

O sistema deve ser executável em diferentes plataformas, como web e desktop, utilizando o framework Flutter.

* RNF02 — Usabilidade

A interface deve ser intuitiva, organizada e de fácil navegação, adequada ao público universitário.

* RNF03 — Segurança

O sistema deve utilizar mecanismos seguros de autenticação, por meio da integração com o Firebase Authentication.

* RNF04 — Manutenibilidade

O código deve ser estruturado de forma modular, facilitando sua manutenção, atualização e expansão.

* RNF05 — Disponibilidade

O sistema deve estar disponível em ambiente web, permitindo acesso remoto por meio de link público.

As telas de autenticação, navegação por categorias, listagem de produtos, visualização de detalhes, carrinho de compras e finalização de pedidos foram efetivamente desenvolvidas, demonstrando coerência entre a modelagem dos requisitos e sua aplicação prática.
